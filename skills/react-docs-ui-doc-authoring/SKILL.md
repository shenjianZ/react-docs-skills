# React Docs UI 文档编写

当用户询问如何在生成后的 docs 项目里编写、组织、维护 Markdown 或 MDX 文档时，使用这个 skill。

默认假设用户正在一个生成后的 docs 项目中工作，例如 `D:\tmp\test-docs\docs`。

## 适用场景

- 用户想知道文档文件应该放在哪里
- 用户想写 Markdown 页面、frontmatter 或 MDX 页面
- 用户想知道自定义组件在 MDX 里怎么用
- 用户想知道中英文文档如何保持一致
- 用户想知道为什么新建了页面文件但导航或侧边栏没有显示
- 用户想新增或维护 `public/docs/<lang>/changelog/` 下的发布记录

不要用这个 skill 回答具体配置字段语义；那类问题交给 `react-docs-ui-config`。

## 工作流程

1. 先判断用户要写的是普通 Markdown、带 frontmatter 的 Markdown、MDX，还是 changelog / release notes。
2. 根据目标 URL 推导应该落到 `public/docs/<lang>/` 下的哪个文件；如果是 changelog，优先落到 `public/docs/<lang>/changelog/<slug>.md` 或 `.mdx`。
3. 如果页面还要出现在导航或侧边栏里，明确提醒还要同步修改配置文件；不要把 changelog 和普通文档页混为一谈。
4. 如果是 changelog，提醒文件名就是最终路由里的 slug，并补齐 frontmatter 中的 `title`、`summary`、`date`、`version`、`type`、`draft`、`breaking` 等常用字段。
5. 如果 changelog 列表页依赖索引文件，明确提醒用户在新增、重命名或删除发布文件后重新生成 changelog 索引。
6. 如果涉及 MDX 组件，说明组件通常放在 `src/components/`，必要时再结合配置处理。
7. 如果项目是双语站点，默认采用"先中文、后英文"的流程：先完成并审核 `zh-cn` 文档，再继续编写 `en` 文档。

## 输出约束

- 回答保持面向最终用户。
- 建议要紧贴 docs 项目的目录结构和使用方式。
- frontmatter 优先使用 `title`、`description`、`author`、`createdAt`、`lastUpdated` 这些字段。
- 编写 changelog 时，优先补齐 `title`、`summary`、`date`、`version`、`type`、`draft`、`breaking`。
- 如果 frontmatter 里已经写了 `title`，要提醒用户避免在正文里重复展示同一个标题。
- 如果用户在模板项目里维护 changelog，要提醒执行 `pnpm prepare:docs` 重新生成索引。
- 默认优先产出中文 Markdown 或 MDX 文档；只有在中文内容审核没有问题后，再继续产出英文版本。英文文档应基于已确认的中文文档做等价翻译，不要先独立编写英文稿。
- 不要承诺"只建文件就会自动进入侧边栏"。
